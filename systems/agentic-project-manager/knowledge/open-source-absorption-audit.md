# Open-Source Absorption Audit

Use this audit before turning any public repository into local Codex workflow knowledge, scripts, skills, or reusable patterns.

## Purpose

Open-source absorption means studying a repository as a source of ideas, patterns, APIs, workflows, or tooling decisions. It is not permission to clone into the workflow repo, copy raw code, install dependencies, or run risky integrations. Safe source-backed knowledge may be AI-audited, activated, validated, redacted, synced, committed, and pushed automatically.

## Audit Checklist

- [ ] Retrieve local workflow knowledge first.
- [ ] Confirm whether this is a source reference, tool candidate, pattern source, script idea, or capability-pack source.
- [ ] Inspect official repository README/docs.
- [ ] Inspect license and attribution requirements.
- [ ] Inspect current activity, releases, issues, and maintenance status when relevant.
- [ ] Identify useful ideas without copying full source or proprietary assets.
- [ ] Map useful parts to existing Codex architecture.
- [ ] Decide whether the result should be a safe blob, safe pack, doc update, script candidate, MCP note, restricted policy, or no action.
- [ ] AI-audit source quality, license boundary, copied-content risk, specs, artifacts, verification, registry metadata, routing fit, and safety boundary.
- [ ] Auto-activate safe knowledge if audit, validation, and redaction pass.
- [ ] Keep generated reports local/task-scoped unless sanitized.
- [ ] Ask explicit approval before installing, cloning large repos, copying source/assets, configuring MCPs, adding keys, enabling paid/session/stealth/account sources, downloading heavy models, running private OCR, enabling services/proxies/wrappers, importing offensive procedures, changing AGENTS.md, or changing production/server/database/auth/deployment behavior.

## Stop Conditions

Stop and report instead of implementing when:

- License is missing, unclear, restrictive, or incompatible.
- The repo requires credentials, private services, production access, or unsafe setup.
- The repo duplicates existing active local capability.
- The useful output is only vague inspiration, not a reusable pattern.
- User approval is needed for install, clone, dependency, MCP, config, execution, sensitive source, external code/assets, or production/server/database/auth/deployment behavior.

## Output

Create a repo absorption report and source absorption ledger entry before proposing any workflow change.
